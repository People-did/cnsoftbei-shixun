# 各格式详细解析策略

本文档定义了各类实训成果文件的详细解析方法、提取要点和质量评估指标。

---

## 1. 文档类（.docx / .doc / .pdf / .txt / .md）

### 1.1 章节结构提取

**目标：** 识别文档的组织结构，判断逻辑是否清晰。

**方法：**
- 提取所有标题（通过字体大小、加粗、编号模式识别层级）
- 构建文档目录树（最多 3 级）
- 检查是否包含标准学术/技术文档的必要章节

**按文档类型检查必要章节：**

| 文档类型 | 必要章节 | 加分章节 |
|---------|---------|---------|
| 实训报告 | 引言/背景、实施过程、结果分析、总结 | 参考文献、附录、代码清单 |
| 项目设计文档 | 需求分析、系统设计、技术选型、实现方案 | 架构图、接口文档、部署说明 |
| 实验报告 | 实验目的、实验方法、实验数据、结论 | 误差分析、对比实验 |
| 毕业设计论文 | 摘要、绪论、相关工作、方法、实验、结论 | 致谢、攻读学位期间成果 |
| 技术方案 | 背景、方案对比、推荐方案、实施计划 | 风险分析、成本估算 |
| 通用总结 | 背景、主要内容、收获/结论 | 不足与展望 |

### 1.2 内容质量评估

**深度评估：**
- 核心论点的独创性和深度
- 技术方案描述的详细程度（是否停留在概念层面，还是有具体实现细节）
- 数据/论据的充分性
- 是否引用了相关文献/资料
- 分析是否到位（不仅描述"做了什么"，还解释"为什么这样做"）

**完整性评估标记：**
- `完整`：含引言、正文（>= 3 个核心章节）、结论，逻辑连贯
- `基本完整`：缺少一个非核心章节，或某章节内容偏少
- `部分缺失`：缺少引言或结论，或核心章节不足 2 个

### 1.3 字数与格式

- 统计正文字数（排除目录、参考文献、代码块、纯引用）
- 检查格式规范：标题编号一致性、图表编号、页眉页脚、引用格式
- 加分项：含图表目录、索引

## 2. 截图类（.png / .jpg / .jpeg / .gif / .bmp / .webp）

### 2.1 截图类型识别

**识别决策树：**

```
截图内容判断
├── 包含窗口标题栏/菜单栏 → 软件/系统界面
│   ├── 含代码编辑器 → 代码截图（归类为代码类）
│   ├── 含数据库管理工具 → 数据库/终端
│   └── 含一般应用界面 → 软件/系统界面
├── 包含终端/命令行 → 数据库/终端
├── 包含大量文本（非UI）→ 运行结果/输出
├── 包含流程图元素（箭头/方框/菱形）→ 流程图/架构图
└── 其他 → 通用截图
```

### 2.2 各类截图解析要点

**软件/系统界面：**
- 提取：窗口标题、菜单项、按钮标签、表单字段、数据表格列名
- 判断：当前处于什么操作阶段（登录、列表、编辑、结果展示）
- 评估：界面布局是否合理、信息展示是否完整

**运行结果/输出：**
- 提取：运行状态标识（success/error/warning）、输出数值、性能指标
- 如有报错信息，完整摘录
- 判断：结果是预期结果还是异常结果

**代码截图：**
- 注意：代码截图的文字需要 OCR 级别的识别，可能不完整
- 如能识别代码，提取语言类型和核心逻辑
- 如代码行数较多（>30行），建议用户同时提交源码文件

**数据库/终端：**
- 提取：执行的命令、SQL 语句、返回结果
- 判断：操作是否成功执行、结果是否符合预期

**流程图/架构图：**
- 提取：节点名称、连线关系、层次结构
- 判断：架构设计是否合理、流程是否完整

### 2.3 多张截图的逻辑串联

1. 优先按文件名排序（识别数字序号）
2. 若文件名无序，按修改时间排序
3. 根据截图内容的上下文关系推断操作顺序
4. 输出操作流程图：`步骤1 → 步骤2 → ... → 步骤N`

### 2.4 质量评估

- **分辨率**：能否清晰辨认文字和界面元素
- **覆盖度**：是否覆盖了核心功能的完整操作流程
- **标注**：截图是否有红框/箭头等标注辅助说明

## 3. 代码类（.py / .java / .js / .ts / .html / .css / .sql / .cpp / .go / .rs 等）

### 3.1 代码文件分析流程

**单个代码文件分析：**

1. **语言识别**：通过文件扩展名和代码特征确认编程语言
2. **结构提取**：
   - 导入/依赖声明
   - 类/接口定义
   - 函数/方法声明
   - 全局变量/常量
3. **规模统计**：总行数、有效代码行数（排除空行和纯注释）、注释行数
4. **质量检查**：
   - 命名规范（camelCase/snake_case 一致性）
   - 注释覆盖率（关键函数是否有文档注释）
   - 错误处理（try-catch、错误返回）
   - 代码复杂度（嵌套深度、函数长度）

**多个代码文件（项目）分析：**

1. **目录结构分析**：
   - 项目根目录识别（通过 package.json/pom.xml/requirements.txt 等）
   - 模块划分识别（src/lib/test/docs 等标准目录）
2. **入口文件定位**：
   - Python: `main.py`, `app.py`, `__init__.py`, `manage.py`
   - Java: `Main.java`, `Application.java`, `*Application.java`
   - JavaScript/TypeScript: `index.js`, `app.js`, `main.ts`
   - Web: `index.html`, `App.vue`, `app.tsx`
3. **技术栈识别**：
   - 从依赖文件中提取：package.json, pom.xml, requirements.txt, go.mod, Cargo.toml, build.gradle
   - 从代码 import 语句中提取

### 3.2 代码质量评估量表

| 维度 | 优秀（90-100%） | 良好（75-89%） | 一般（60-74%） | 较差（<60%） |
|------|---------------|---------------|---------------|-------------|
| 代码结构 | 模块划分清晰，单一职责 | 模块划分基本合理 | 部分耦合过度 | 结构混乱 |
| 命名规范 | 语义化命名，风格统一 | 大多数命名合理 | 存在无意义命名 | 大量 a/b/c 等 |
| 注释质量 | 关键逻辑完整注释 | 主要函数有注释 | 注释稀疏 | 无注释或过期注释 |
| 错误处理 | 完善的异常处理 | 核心逻辑有处理 | 只有基础处理 | 无错误处理 |
| 代码复用 | 无明显重复 | 少量重复 | 中等重复 | 大量复制粘贴 |

### 3.3 文件数量与解析策略

| 文件数量 | 解析策略 |
|---------|---------|
| 1-5 个 | 全部详细解析 |
| 6-20 个 | 详细解析入口 + 核心模块（前 10 个），其余简要列出 |
| 21-50 个 | 详细解析入口 + README，核心模块简要评估，其余仅列文件名 |
| > 50 个 | 仅解析入口文件 + README，按目录汇总其余文件 |

## 4. ZIP 压缩包

### 4.1 解压与文件树输出

1. 使用 `scripts/extract_zip.py` 解压到临时目录
2. 输出前 3 层文件树结构
3. 统计文件类型分布

### 4.2 项目类型识别

**根据文件特征识别项目类型：**

| 特征文件/目录 | 识别项目类型 |
|-------------|------------|
| `package.json` + `node_modules/` | Node.js / 前端项目 |
| `pom.xml` + `src/main/java/` | Java Maven 项目 |
| `build.gradle` / `settings.gradle` | Java Gradle 项目 |
| `requirements.txt` / `setup.py` / `pyproject.toml` | Python 项目 |
| `go.mod` / `go.sum` | Go 项目 |
| `Cargo.toml` | Rust 项目 |
| `composer.json` | PHP 项目 |
| `.sln` / `.csproj` | .NET 项目 |
| `Dockerfile` + 多个服务目录 | 容器化/微服务项目 |
| `index.html` + `css/` + `js/` | 静态 Web 项目 |
| `vue.config.js` / `vite.config.js` | Vue 前端项目 |
| `angular.json` | Angular 项目 |
| `next.config.js` | Next.js 项目 |
| `.ipynb` 文件集合 | Jupyter Notebook 项目 |
| SQL 文件集合 | 数据库设计项目 |
| 仅含文档和截图，无代码 | 文档/设计类成果 |

### 4.3 ZIP 内多项目处理

若 ZIP 根目录下存在多个独立项目目录（各自有独立的构建文件），则：
1. 识别所有子项目
2. 选择文件数量最多/规模最大的作为主项目进行详细评价
3. 其余项目在报告附录中简要列出
4. 提示："检测到压缩包内含多个项目，以下为主项目的评价结果"

## 5. 混合成果处理

当用户上传了多种类型的文件时：

1. **分类归组**：将文件按类型分组（文档/截图/代码）
2. **主成果判定**：
   - 既有文档又有代码 → 以文档为主，代码为支撑
   - 既有截图又有文档 → 以文档为主，截图为佐证
   - 仅有截图和代码 → 以代码为主，截图为运行演示
3. **综合评价权重**：按主成果类型分配评分权重
4. **关联分析**：
   - 文档中描述的功能是否在代码中实现
   - 截图中展示的界面是否与文档描述一致
   - 代码的运行结果是否与截图一致

---

## 6. 无法识别的文件格式

| 情况 | 处理 |
|------|------|
| `.exe` / `.dll` / `.so` | 跳过，注明"二进制文件，无法解析" |
| `.pptx` / `.xlsx` | 如有对应工具尝试提取文字，否则提示用户导出为 PDF 后上传 |
| `.rar` / `.7z` / `.tar.gz` | 提示用户转换为 `.zip` 格式 |
| `.mp4` / `.avi` | 视频文件暂不支持，请用户上传关键截图 |
| 无扩展名文件 | 尝试用 `file` 命令检测类型，若无法识别则跳过 |
