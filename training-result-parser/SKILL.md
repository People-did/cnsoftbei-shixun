# 实训成果智能解析与评价 Skill

面向大学生实训系统，支持多格式成果文件上传，由 AI 自动解析内容、提取核心信息，
并依据评分标准生成结构化评价报告。科目类型通用，适用于任意专业方向的实训作业。

## 触发词

当用户说出以下任意词语时触发本 Skill：

- "解析实训成果" / "分析上传的文件" / "帮我看看这份实训报告"
- "解析作业文件" / "评价实训成果" / "批改实训作业"
- "分析截图" / "解析上传成果" / "实训评分"
- "解析这份报告" / "帮我评分" / "实训成果评价"

---

## 支持的上传格式

| 格式 | 说明 |
|------|------|
| `.docx` / `.doc` | 实训报告、项目说明、设计文档 |
| `.pdf` | 实训总结、论文、方案报告 |
| `.txt` / `.md` | README、开发日志、纯文本说明 |
| `.png` / `.jpg` / `.jpeg` | 软件界面截图、运行结果截图、操作流程截图 |
| `.py` / `.java` / `.js` / `.ts` / `.html` / `.css` / `.sql` 等 | 源代码文件 |
| `.zip` | 含源码或多文件的项目压缩包 |

---

## 工作流程（严格按此顺序执行）

### ⚠️ 关键顺序：先询问评分标准 → 再解析文件 → 最后对照标准输出评价报告

在任何情况下都不得跳过第一步直接进入文件解析。即使用户在触发消息中已经描述了评分标准，也必须先确认是否完整、是否还有补充。

---

### 【第一步：评分标准确认】—— 最高优先级，必须首先执行

在读取任何上传文件的内容之前，必须先向用户确认评分标准。

**输出以下提示：**

> 📋 **评分标准确认**
>
> 请问您是否有指定的评分标准（Rubric）？
>
> - **如有**：请直接粘贴或描述评分维度及分值；
> - **如无**：我将根据上传成果的内容类型，自动生成评分标准。
>
> 请告知您的选择，或直接提供评分标准。

**处理逻辑：**

1. **用户提供了评分标准**（粘贴文本 / 描述维度与分值 / 上传评分标准文件）：
   - 解析用户提供的评分标准，提取：维度名称、分值、评分要点
   - 记录到工作记忆中，供第三步使用
   - 简要确认："已记录您提供的评分标准（共 N 个维度，满分 100 分），将依此进行评价。"
   - 跳转至第二步

2. **用户明确表示无评分标准**（"没有" / "你定" / "自动生成" 等）：
   - 回复："好的，将在解析完上传文件后，根据成果类型自动生成评分标准。"
   - 跳转至第二步（解析完成后自动生成）

3. **用户在触发消息中已附带了评分标准**：
   - 确认是否完整，向用户复述解析结果
   - 询问："以上是我从您的描述中提取的评分标准，是否准确？如需调整请告知，否则将按此标准评价。"
   - 用户确认后，跳转至第二步

**自动生成评分标准的规则**（当用户未提供时，在第二步解析完成后触发）：

| 成果类型 | 默认评分维度与分值 |
|---------|-------------------|
| 文档报告类 | 内容完整性 30 + 逻辑结构 25 + 技术深度 30 + 格式规范 15 |
| 代码/程序类 | 功能实现正确性 40 + 代码结构规范 20 + 注释与可读性 15 + 文档完整性 25 |
| 截图操作类 | 操作步骤完整 35 + 功能覆盖度 35 + 界面规范度 20 + 说明清晰度 10 |
| 混合成果（文档+代码+截图）| 文档质量 30 + 代码质量 30 + 功能演示 25 + 整体完整度 15 |
| 无法识别类型 | 完整性 35 + 规范性 30 + 核心内容质量 35 |

自动生成后，向用户展示并说明：

> 已根据成果类型「xxx」自动生成以下评分标准，将依此进行评价，如需调整请告知：
> （列出维度与分值表格）

---

### 【第二步：文件解析】

按用户提供的文件路径，逐一读取和解析。根据文件扩展名选择对应的解析策略。

#### ① 文档类文件（`.docx` / `.doc` / `.pdf` / `.txt` / `.md`）

**对于 `.docx` / `.doc` 文件：**
1. 使用 `python -c "import docx; ..."` 或类似方式提取文本内容
2. 若 Python 环境不可用，使用 `antiword` 或提示用户安装 `python-docx`
3. 若以上均不可用，使用 `strings` 或 `catdoc` 提取文本（可能丢失格式）

**对于 `.pdf` 文件：**
1. 优先使用 `pdftotext` 提取文本
2. 若无 `pdftotext`，尝试 `python -c "import PyPDF2; ..."` 或 `pdfminer`
3. 若 PDF 为扫描版（图片），说明无法提取文字，请用户提供文字版或清晰的截图

**对于 `.txt` / `.md` 文件：**
1. 直接使用 Read 工具读取内容

**文档解析要点：**
- 提取章节结构（标题层级、目录）
- 提取核心论点、技术方案、实现描述、结论
- 统计字数，判断文档完整度（是否含引言/正文/结论等必要部分）
- 识别文档主题和成果类型（如：文献综述、方案设计、实验报告、项目总结等）
- 提取关键技术术语和核心概念

#### ② 截图文件（`.png` / `.jpg` / `.jpeg`）

1. 使用 Read 工具读取图片内容（支持视觉识别）
2. 识别截图类型：
   - **软件/系统界面** → 提取功能模块名称、UI 布局、操作按钮、交互流程
   - **运行结果截图** → 提取运行状态（成功/报错信息）、输出数据、性能指标
   - **代码截图** → 提取语言类型、核心逻辑片段、变量/函数名
   - **数据库/终端截图** → 提取命令语句、执行结果、数据表结构
   - **流程图/架构图** → 提取节点、连线关系、架构层次
3. 多张截图时，按以下优先级梳理逻辑顺序：
   - 文件名中的序号（如 step1.png, step2.png）
   - 截图内容中的操作先后关系
   - 文件修改时间排序

**截图解析输出格式：**
- 截图数量
- 按逻辑顺序描述操作内容
- 评估功能覆盖度（是否覆盖了核心功能的演示）
- 标注无法识别的部分（如图片模糊、分辨率不足等）

#### ③ 代码文件（`.py` / `.java` / `.js` / `.ts` / `.html` / `.css` / `.sql` 等）

1. 使用 Read 工具读取源文件内容
2. 提取信息：
   - 编程语言及版本特性
   - 项目/模块整体结构
   - 核心函数/类/方法列表及用途
   - 技术栈（依赖的库/框架）
   - 代码规模（行数、文件数）
3. 评估维度：
   - 代码结构合理性（模块划分、职责分离）
   - 命名规范（变量/函数/类命名是否语义化）
   - 注释覆盖情况（关键逻辑是否有注释说明）
   - 错误处理（异常捕获、边界条件）
   - 代码复用（是否存在重复代码块）

**代码文件超过 20 个时：**
- 仅详细解析主入口文件（如 `main.py`、`App.java`、`index.js`）和核心业务模块
- 其余文件仅列出文件名和行数概要

#### ④ ZIP 压缩包

1. 使用 `scripts/extract_zip.py` 脚本解压并列出文件树：
   ```
   python scripts/extract_zip.py <zip文件路径>
   ```
2. 脚本输出前 3 层目录结构的文件树
3. 若第 3 层仍有子目录，汇总统计深层文件数量
4. 根据解压后的文件类型，分别按文档/截图/代码的策略处理核心文件
5. 文件数超过 50 个时，仅解析入口文件和 README（如有），其余列出清单

**详细策略见：** [references/file-type-strategies.md](references/file-type-strategies.md)

---

### 【第三步：生成评价报告】

对照评分标准（用户提供或自动生成），逐维度打分并给出具体依据。

**输出前确认：**
- 评分标准是否已确认（用户提供或自动生成并展示）
- 所有文件是否已解析完毕
- 各维度的评分依据是否有文件原文支撑

### 评分原则

1. **依据必须具体**：每条评分的"评分说明"必须引用文件中的原文、截图中的具体元素或代码中的具体函数/类作为依据，不得仅凭主观判断
2. **扣分要说明原因**：每处扣分必须说明具体扣在哪里、为什么扣、扣了多少
3. **满分也要有依据**：满分项同样需要说明为什么值得满分（具体哪些表现好）
4. **整体评价要平衡**：既要指出优点给予肯定，也要明确指出不足

## 边界情况处理

| 情况 | 处理方式 |
|------|---------|
| 文件无法读取（损坏/权限/格式不支持） | 报告中注明该文件跳过，继续处理其他文件；不影响整体评分 |
| 仅上传截图无文档 | 完成截图解析，在报告末尾注明："⚠️ 建议同时提交书面报告以便完整评价" |
| 图片清晰度不足/模糊 | 说明无法识别的具体部分，请用户补充文字说明；该部分不强制扣分 |
| 压缩包层级过深（>3层） | 只解析前 3 层结构，汇总其余文件为文件清单 |
| 代码文件超过 20 个 | 仅详细解析主入口和核心模块（最多 10 个核心文件），其余汇总列出 |
| 代码文件超过 50 个 | 仅解析入口文件 + README，其余仅列表 |
| 上传文件格式全部无法识别 | 提示用户支持的格式列表，请求重新上传 |
| ZIP 中包含多个独立项目 | 分别识别，按主要项目评价，次要项目在附录中简要列出 |
| 文档语言非中文 | 正常解析英文/混合内容，评分标准不变 |
