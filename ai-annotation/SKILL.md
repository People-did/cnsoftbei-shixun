# AI智能批注 Skill

面向大学生实训系统，AI自动读取学生提交的实训报告原文，像真人教师一样逐段审阅，
找出具体问题，在原文中精确定位并生成批注（划线标注 + 问题说明 + 修改建议）。

## 触发词

当用户说出以下任意词语时触发本 Skill：

- "AI批注" / "智能批注" / "AI批改标注"
- "批注这份报告" / "帮我批注一下" / "逐段批注"
- "给这份作业划线批注" / "标注问题" / "AI划线批改"
- "智能标注问题" / "逐段分析并标注"

---

## 支持的报告格式

| 格式 | 说明 |
|------|------|
| 纯文本报告 | 存储在 `training_report.content` 字段中的文本内容 |
| Word文档 | 通过系统已集成的 Apache POI 提取文本后批注 |
| PDF文档 | 通过系统已集成的 PDFBox 提取文本后批注 |
| 代码文件 | 对源码进行逐行/逐段的代码质量批注 |

## 工作流程

### 【第一步：获取报告内容】

1. 确定要批注的报告ID（`reportId`）
2. 通过 `GET /api/report/{id}` 获取报告完整内容（`content` 字段）
3. 如果报告内容为空，提示用户"报告内容为空，无法生成批注"
4. 如果报告内容过长（超过8000字），截取前8000字发送给AI

### 【第二步：AI逐段分析并生成批注】

调用 `POST /api/evaluation/annotate/{reportId}` 触发AI批注。

AI将按照以下维度逐段审查报告：

| 批注维度 | 检查内容 | 严重程度示例 |
|---------|---------|-------------|
| **规范性** | 格式是否规范，命名是否合理，文档结构是否清晰 | warning / info |
| **完整性** | 是否有明显的章节缺失，核心内容是否充分 | error / warning |
| **知识点** | 技术表述是否准确，概念使用是否正确 | error / warning |
| **代码质量** | 代码片段是否规范，逻辑是否正确，是否有明显bug | error / warning |
| **文档结构** | 章节组织是否合理，逻辑是否连贯 | info / suggestion |
| **逻辑问题** | 论述是否有前后矛盾、逻辑漏洞 | error |
| **表述问题** | 语句是否通顺，用词是否准确，是否存在语病 | info / suggestion |

严重程度分级：

| 级别 | 标识 | 含义 | 前端颜色 |
|------|------|------|---------|
| `error` | 🔴 严重 | 知识性错误、逻辑硬伤、关键功能遗漏 | 红色高亮+波浪线 |
| `warning` | 🟠 需改进 | 格式不规范、表述不清晰、内容不充分 | 橙色高亮+虚线 |
| `info` | 🔵 建议 | 可优化的地方、补充说明的建议 | 蓝色高亮 |
| `suggestion` | 🟢 优化 | 锦上添花的改进方向 | 绿色高亮 |

### 【第三步：展示批注结果】

批注结果以双栏布局展示：

- **左侧**：报告原文，批注段落以不同颜色背景高亮
  - 鼠标悬停高亮文字 → 弹出Popover显示批注详情
  - 点击高亮文字 → 右侧面板滚动到对应批注卡片
- **右侧**：批注卡片列表，按原文顺序排列
  - 点击批注卡片 → 原文自动滚动到对应高亮位置
  - 每张卡片显示：严重程度标签、问题分类、原文引用、问题说明、修改建议

## AI批注Prompt模板

```
你是一位资深的软件工程教师，正在逐段仔细批改一份学生实训报告。
你的任务不是给整体分数，而是像真人教师改作业一样——找出报告中每一个需要批注的具体问题，
并给出精确的原文标注和修改建议。

## 批注维度
请从以下维度审视报告内容：
1. **规范性**：格式是否规范，命名是否合理，文档结构是否清晰
2. **完整性**：是否有明显的章节缺失，内容是否充分
3. **知识点**：技术表述是否准确，概念使用是否正确，是否有知识性错误
4. **代码质量**：代码片段是否规范，逻辑是否正确，是否有明显bug
5. **文档结构**：章节组织是否合理，逻辑是否连贯
6. **逻辑问题**：论述是否有前后矛盾、逻辑漏洞
7. **表述问题**：语句是否通顺，用词是否准确，是否存在语病

## 输出格式（纯JSON数组，不要包含任何markdown标记或其他文字）
[
  {
    "highlightedText": "报告中需要批注的原文（请精确摘录，至少15个字符，确保可在原文中唯一定位）",
    "category": "规范性",
    "severity": "error",
    "comment": "具体的问题说明，指出这里为什么有问题（50-150字）",
    "suggestion": "具体的修改建议，给出可操作的修改方式（50-200字）"
  }
]

## 批注要求
1. 只标注确实存在问题的内容，不要过度批注（每1000字控制在5-8条）
2. highlightedText 必须是原文中的精确文字，不要改写
3. 每条批注的comment要具体，必须引用原文细节说明问题
4. suggestion要给出可操作的修改方案，不要空话套话
5. 优先批注严重问题（error），其次是一般问题（warning），最后是建议（info/suggestion）
6. 同一段文字如果同时有多个问题，请分别列出
7. 如果原文确实找不到明显问题，返回空数组 []
```

## 边界情况处理

| 情况 | 处理方式 |
|------|---------|
| 报告内容为空 | 返回错误提示"报告内容为空，无法生成批注" |
| AI API调用失败 | 前端显示空状态，提示"AI批注服务暂时不可用"并提供重试按钮 |
| 批注文本在原文中无法定位 | 该条批注的 startPos/endPos 保持为空，前端不显示高亮但仍在卡片列表中以"浮动批注"形式展示 |
| 多条批注重叠 | 按严重程度优先级（error > warning > info > suggestion）保留高优先级的，裁剪低优先级的重叠区域 |
| 同一报告再次触发批注 | 先删除旧批注，再插入新批注（覆盖逻辑） |
| AI未发现明显问题 | 返回空列表，前端显示"AI未发现明显问题，报告质量较好！" |
| 报告内容超过8000字 | 截取前8000字发送给AI，避免token超限 |

## 与其他功能的配合

- **AI评价**：先进行AI智能评分（整体打分），再进行AI智能批注（逐段划线），两者互补
- **AI对话反馈**：学生可就批注中的具体问题与AI进行多轮对话
- **教师人工修改**：教师在查看AI批注后，可手动修改评分或添加评语

## 数据库表结构

批注数据存储在 `report_annotation` 表中：

```sql
CREATE TABLE report_annotation (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  report_id BIGINT NOT NULL,
  start_pos INT DEFAULT 0,
  end_pos INT DEFAULT 0,
  highlighted_text TEXT,
  comment TEXT,
  severity VARCHAR(20) DEFAULT 'info',
  category VARCHAR(50),
  suggestion TEXT,
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  KEY idx_report_id (report_id)
);
```

## API端点

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/evaluation/annotate/{reportId}` | 触发AI智能批注（教师/管理员） |
| `GET` | `/api/evaluation/annotations/{reportId}` | 获取已保存的批注列表 |
| `DELETE` | `/api/evaluation/annotations/{reportId}` | 删除批注（教师/管理员） |
